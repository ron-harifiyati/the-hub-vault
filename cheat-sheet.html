Cheat Sheet

The Fixer

Plain-English definitions for every GitHub term, plus solutions to the most common iPad problems. Bookmark this page.

Glossary

Blunt Truths

No jargon. No fluff. Here is what these words actually mean.

📁 Repository
A project folder on GitHub. Everything related to one project lives here — presentations, reports, images, notes. One project = one repository.
💾 Commit
A permanent Save Point. When you commit, GitHub records what you uploaded, the message you wrote, and the exact time it happened. It cannot be undone silently — the history is always there.
📖 README
The front cover of your repository. It is the first thing anyone sees when they open your project. Write what the project is and why it matters. Two sentences is enough to start.
🕐 Commit History
The full record of every save you have ever made in a repository — with timestamps. This is your proof of consistent, disciplined work. It cannot be faked.
🌿 Branch
A parallel version of your project where you can try something new without breaking the original. Advanced feature — students don't need this yet, but it's worth knowing the word exists.
🔀 Merge
Combining a branch back into the main project. Again — advanced. You'll encounter the word. Now you know what it means.
🐛 Issue
A built-in comment or task card on a repository. Facilitators use Issues to leave feedback on student work. Think of it as a sticky note attached to the project.
⭐ Star
GitHub's equivalent of a "like." When someone stars your repository, it means they found it valuable. More stars = more visibility in the GitHub community.
🍴 Fork
Making your own copy of someone else's repository so you can modify it without changing the original. Useful later — ignore it for now.
🔒 Private vs Public
Public = anyone on the internet can see it. Private = only you (and people you invite) can see it. Hub repositories should be Public. Your personal experiments can be Private.
📤 Push
Sending your local changes up to GitHub. When you're using GitHub in Safari on iPad, uploading IS pushing — the browser handles it for you automatically.
🏠 Main / Master
The primary branch of a repository — the "real" version. When you upload directly from Safari, your files go here by default. Nothing to worry about at this stage.

iPad Troubleshooting

When Things Go Wrong

These are the most common problems at the hub. Check here before asking for help.

😵 Safari "Ghosting" — The Upload Button Doesn't Respond Common

Sometimes Safari on iPad appears frozen — you tap a button and nothing happens.

Fix:

  1. Tap the address bar in Safari
  2. Pull down on the page to refresh, or tap the reload icon (↺)
  3. Try the action again
  4. If still frozen: close Safari completely (swipe up from the bottom, swipe the Safari preview away) and reopen it
[IMAGE: iPad Safari with the reload/refresh button circled in the address bar]

📦 File Too Big to Upload Common

GitHub has a 25MB limit per file when uploading via browser. A Keynote exported at "Best" quality can easily exceed this.

Fix:

  1. Go back to Keynote and re-export as PDF
  2. When choosing quality, select Good instead of Best
  3. This reduces file size significantly with minimal visual difference
  4. If still too large: reduce the number of slides per file, or remove large images from the presentation
⚠️ Remember "Best" quality Keynote exports can be 50MB+. "Good" quality is typically under 10MB. For GitHub uploads, Good is always sufficient.

🔑 Login Issues / Locked Out of GitHub Important

If you forget your password or lose access to your account, GitHub uses Recovery Codes to verify your identity.

Prevention (do this now, before it happens):

  1. Go to GitHub Settings → Password and Authentication
  2. Find your Recovery Codes and download or copy them
  3. Write them in a physical notebook kept at the hub — not just on your iPad
  4. Never store your only copy of recovery codes on the same device you use to log in
⚠️ iPads Get Wiped If your recovery codes are only saved on your iPad and the iPad gets wiped, you may lose access to your account permanently. Write. Them. Down.

🗂️ Can't Find the File in the Files App Common

After exporting from Keynote, students sometimes can't find the PDF when trying to upload to GitHub.

Fix:

  1. Open the Files app on your iPad
  2. Check On My iPad → Keynote — exports often save here by default
  3. Also check Recents at the top of the Files app — it shows your most recently saved files
  4. If you saved to iCloud, make sure iCloud Drive is turned on in Settings
[IMAGE: iPad Files app showing the "Recents" section with a recently exported PDF visible]

🌐 GitHub Looks Different Than Expected Minor

GitHub occasionally updates its interface. If the screenshots in this guide look slightly different from what you see, the functionality is still the same — only the layout may have shifted.

What to look for instead of exact button locations:

  • The + icon or Add file button to upload
  • The Commits link near the file list to see history
  • The Issues tab at the top of any repository
  • The Edit (pencil) icon to edit the README

📵 No Internet at the Hub Situational

GitHub requires an internet connection. If the hub's connection is down, you can still do the work — just save it for later.

Offline workflow:

  1. Continue working in Keynote — you don't need internet for that
  2. Export your PDF and save it to the Files app as normal
  3. When the connection is restored, upload everything you saved
  4. Use a descriptive commit message that mentions the actual date of the work

Quick Reference

The Workflow at a Glance

Student Weekly Workflow

  1. Work in Keynote on iPad
  2. Tap ··· → Export → PDF → Good quality
  3. Tap Save to Files with a clear filename
  4. Open Safari → go to your GitHub repo
  5. Tap Add file → Upload files
  6. Select your PDF from Files
  7. Write a commit message describing what it is
  8. Tap Commit changes ✅

Facilitator Friday Workflow

  1. Write weekly report in Keynote or Pages
  2. Export as PDF → Good quality
  3. Name it: Week-[N]-[HubName]-Report.pdf
  4. Upload to reports/ folder in hub repository
  5. Write a commit message with student count and highlights
  6. Tap Commit changes
  7. Send sponsor the repo URL — not an attachment ✅
